# Лабораторная работа 2

## Индексы, транзакции, уровни изоляции и Cache Aside

Эта лабораторная работа демонстрирует несколько важных возможностей PostgreSQL и Redis. После запуска контейнер `app` выполняет Python-скрипт `lab2.py`, последовательно прогоняет все сценарии и печатает результаты в логи.

## Что запускается

- `lab2-db` - PostgreSQL 16, доступен на `localhost:5433`.
- `lab2-redis` - Redis, доступен на `localhost:6380`.
- `lab2-app` - Python-скрипт, который выполняет демонстрацию.

## Что происходит внутри

Скрипт выполняет четыре части.

### 1. Индексы

Создается таблица `users(id, name, email)`, затем в нее добавляется 20 000 записей.

После этого выполняется поиск пользователя по `email`:

1. сначала без индекса;
2. затем создается индекс `idx_users_email`;
3. тот же запрос выполняется повторно;
4. результат сравнивается через `EXPLAIN ANALYZE`.

В логах видно, как меняется план выполнения, время запроса и количество проверенных строк.

### 2. Транзакции и откат

Создается таблица `accounts(id, balance)`. Скрипт имитирует перевод денег между счетами внутри транзакции.

После изменения балансов специально выбрасывается ошибка. PostgreSQL откатывает всю транзакцию, поэтому итоговые балансы остаются такими же, как до перевода.

Эта часть показывает атомарность: операция либо выполняется целиком, либо не выполняется вообще.

### 3. Уровни изоляции

Показываются два сценария:

- `READ COMMITTED` - повторное чтение в одной транзакции может увидеть уже закоммиченные изменения из другой транзакции;
- `REPEATABLE READ` - повторное чтение видит тот же снимок данных, что и при первом чтении.

Так демонстрируется разница между уровнями изоляции и эффект non-repeatable read.

### 4. Cache Aside

Redis используется как внешний кэш.

Логика такая:

1. приложение сначала ищет пользователя в Redis;
2. если данных нет, читает их из PostgreSQL;
3. затем кладет результат в Redis через `setex`;
4. повторный запрос с тем же ключом берет данные уже из кэша.

В логах видно, что количество обращений к базе уменьшается.

## Состав файлов

- `docker-compose.yml` - PostgreSQL, Redis и приложение.
- `app/Dockerfile` - сборка Python-контейнера.
- `app/requirements.txt` - зависимости.
- `app/lab2.py` - основной демонстрационный скрипт.

## Запуск

Из папки `lab_2`:

```bash
docker compose up --build
```

Контейнер `app` выполнит лабораторную работу и завершится. PostgreSQL и Redis останутся запущенными, пока вы не остановите compose.

## Просмотр логов

```bash
docker compose logs app
```

В логах нужно обратить внимание на блоки:

- `PART 1: INDEXES`;
- `PART 2: TRANSACTIONS AND ROLLBACK`;
- `PART 3: ISOLATION LEVELS`;
- `PART 4: CACHE ASIDE`.

Если все прошло успешно, в конце будет сообщение:

```text
All lab scenarios finished successfully.
```

## Остановка

```bash
docker compose down
```

Остановить и удалить данные PostgreSQL/Redis:

```bash
docker compose down -v
```

## Итог

Лабораторная показывает, как индексы ускоряют поиск, как транзакции защищают данные от частичных изменений, чем отличаются уровни изоляции и как Redis помогает снизить количество запросов к базе через паттерн Cache Aside.
