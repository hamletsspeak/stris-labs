# Лабораторная работа 4

## Партиционирование, шардирование и репликация

В этой лабораторной работе проектируется база данных для приложения учета доходов и расходов. Сценарий похож на личный кошелек: есть пользователи, счета, категории, транзакции и дневные агрегаты.

Работа показывает три идеи:

- партиционирование большой таблицы транзакций по дате;
- шардирование пользователей по `user_id`;
- репликацию PostgreSQL внутри каждого шарда.

## Что запускается

В Docker Compose поднимаются два шарда. У каждого шарда есть primary и replica:

- `lab4-shard1-primary` - primary первого шарда, порт `5441`;
- `lab4-shard1-replica` - replica первого шарда, порт `5442`;
- `lab4-shard2-primary` - primary второго шарда, порт `5443`;
- `lab4-shard2-replica` - replica второго шарда, порт `5444`;
- `lab4-app` - Python-скрипт, который создает схему, записывает данные и читает отчеты.

## Схема данных

Основные таблицы:

- `users(id, full_name, email, created_at)`;
- `accounts(id, user_id, account_name, balance)`;
- `categories(id, user_id, category_name, kind)`;
- `transactions(transaction_id, user_id, account_id, category_id, amount, note, transaction_date, created_at)`;
- `daily_user_aggregates(user_id, day, total_income, total_expense)`.

Связи:

```text
users 1:N accounts
users 1:N categories
users 1:N transactions
accounts 1:N transactions
categories 1:N transactions
users 1:N daily_user_aggregates
```

## Партиционирование

Таблица `transactions` партиционируется по полю `transaction_date`.

Используется `RANGE PARTITION`: для каждого месяца создается отдельная партиция вида:

```text
transactions_YYYY_MM
```

Также есть `transactions_default` для дат, которые не попали в заранее созданные диапазоны.

Зачем это нужно:

- запросы за конкретный месяц читают только нужную партицию;
- старые данные проще архивировать или удалять;
- уменьшается объем сканирования при аналитических запросах.

## Шардирование

Пользователи распределяются по двум шардам по правилу:

```text
shard_index = user_id % 2
```

Если остаток `0`, пользователь попадает в `shard_1`. Если остаток `1`, в `shard_2`.

Все данные пользователя лежат в одном шарде: профиль, счета, категории, транзакции и агрегаты. Это упрощает локальные транзакции и уменьшает необходимость межшардовых запросов.

## Репликация

В каждом шарде запись идет в primary, а чтение отчетов выполняется из replica.

Схема:

```text
[Router by user_id % 2]
        |
        |----> [Shard 1 Primary] -> [Shard 1 Replica]
        |
        |----> [Shard 2 Primary] -> [Shard 2 Replica]
```

Такой подход отделяет write-нагрузку от read-нагрузки. Возможный минус - задержка репликации: сразу после записи replica может еще не увидеть новые данные.

## Что делает приложение

Контейнер `app`:

1. ждет готовности всех PostgreSQL-узлов;
2. создает схему на primary-шардах;
3. создает партиции таблицы `transactions`;
4. распределяет тестовых пользователей по шардам;
5. записывает данные в primary;
6. читает отчеты из replica;
7. печатает результат в логи.

## Запуск

Из папки `lab_4`:

```bash
docker compose up --build
```

## Просмотр результата

```bash
docker compose logs app
```

В логах должно быть видно:

- какой пользователь попал в какой шард;
- что схема создана;
- что данные записаны;
- что отчеты прочитаны из реплик;
- сообщение `Lab 4 demo finished successfully.`

## Остановка

```bash
docker compose down
```

Остановить и удалить данные PostgreSQL:

```bash
docker compose down -v
```

## Итог

Лабораторная показывает, как проектировать базу для растущего объема данных: крупные таблицы делятся на партиции, пользователи распределяются по шардам, а реплики помогают разгрузить чтение.
