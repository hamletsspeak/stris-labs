# STRIS Labs

Лабораторные работы по дисциплине **Modern Technologies for the Development of Information Systems**.

Аббревиатура репозитория: **STRIS** / **СТРИС** — современные технологии разработки информационных систем.

## Назначение репозитория

Репозиторий содержит набор лабораторных работ, посвященных проектированию, разработке и развертыванию информационных систем с использованием современных backend-, database- и infrastructure-подходов.

Каждая лабораторная работа оформлена как отдельный проект в собственной директории `lab_N`. Внутри каждой директории находится локальный `README.md` с описанием задания, состава решения, команд запуска и способов проверки.

## О дисциплине

Дисциплина рассматривает технологии и архитектурные подходы, применяемые при создании современных информационных систем. Основной акцент сделан на практическую реализацию компонентов, которые отвечают за API-контракты, обработку HTTP-запросов, хранение данных, масштабирование, отказоустойчивость, асинхронное взаимодействие и инфраструктурные паттерны.

В рамках лабораторных работ изучаются:

- проектирование REST API и документирование контрактов;
- построение HTTP-сервисов и работа через reverse proxy;
- балансировка нагрузки между экземплярами сервиса;
- кэширование данных;
- индексы, транзакции и уровни изоляции в СУБД;
- репликация, шардирование и партиционирование данных;
- событийная обработка через брокер сообщений;
- архитектурные паттерны распределенных web-систем;
- надежность, деградация, повторные попытки и защита от перегрузки.

## Содержание лабораторных работ

### Лабораторная работа 0. REST API и OpenAPI

В директории [`lab_0`](./lab_0) находится спецификация REST API для социальной сети.

Реализовано описание API-контракта в формате OpenAPI 3.0.3. Спецификация определяет операции просмотра анкеты пользователя, добавления и удаления друзей, просмотра списка друзей и публикации постов. Также описаны форматы успешных ответов и ошибок.

Работа отвечает за этап проектирования внешнего API до реализации backend-сервиса. Такой подход позволяет заранее определить структуру эндпоинтов, схемы данных и правила взаимодействия клиента с сервером.

### Лабораторная работа 1. HTTP-сервис, балансировка и кэш

В директории [`lab_1`](./lab_1) реализован HTTP-сервис, запускаемый в нескольких экземплярах.

Система состоит из двух backend-инстансов, Nginx и Redis. Nginx выполняет роль единой точки входа, reverse proxy и балансировщика нагрузки. Запросы к сервису распределяются между экземплярами по алгоритму Round Robin. Redis используется для кэширования результатов запросов к `/data`.

Работа демонстрирует базовую инфраструктуру web-приложения: несколько экземпляров сервиса, единый входной endpoint, распределение нагрузки и уменьшение повторных вычислений за счет кэша.

### Лабораторная работа 2. Индексы, транзакции и Cache Aside

В директории [`lab_2`](./lab_2) находится проект для демонстрации работы PostgreSQL и Redis.

PostgreSQL используется для проверки влияния индексов на выполнение запросов, демонстрации атомарности транзакций и сравнения уровней изоляции `READ COMMITTED` и `REPEATABLE READ`. Redis используется как внешний кэш в паттерне Cache Aside: приложение сначала проверяет наличие данных в кэше, а при промахе читает их из базы данных и сохраняет в Redis.

Работа показывает, как СУБД обеспечивает целостность данных, как индексы влияют на производительность и как кэширование снижает количество обращений к базе.

### Лабораторная работа 3. Распределенное хранение данных

В директории [`lab_3`](./lab_3) реализована простая модель распределенного key-value хранилища.

Система состоит из master-узла и двух replica-узлов. Запись выполняется через master, после чего данные отправляются на реплики. Все значения хранятся в памяти процессов. При остановке одной из реплик и последующей записи через master демонстрируется расхождение состояния между узлами.

Работа отвечает за понимание базовых принципов репликации, возможной потери актуальности данных на репликах и проблем согласованности в распределенных системах.

### Лабораторная работа 4. Партиционирование, шардирование и репликация

В директории [`lab_4`](./lab_4) спроектирована база данных для приложения учета доходов и расходов.

Данные разделяются по нескольким уровням. Таблица транзакций партиционируется по дате, что ускоряет запросы по временным диапазонам и упрощает управление историческими данными. Пользователи распределяются между шардами по `user_id`, что позволяет горизонтально масштабировать хранение. Для каждого шарда предусмотрены primary- и replica-узлы: записи выполняются в primary, чтение отчетов выполняется из replica.

Работа демонстрирует подходы к масштабированию базы данных при росте объема информации и нагрузки.

### Лабораторная работа 5. Kafka и событийная обработка

В директории [`lab_5`](./lab_5) реализована система обмена сообщениями через Kafka.

Producer отправляет JSON-сообщения о транзакциях в topic `transactions`. Topic содержит несколько partitions. Ключом сообщения является `user_id`, поэтому события одного пользователя попадают в одну и ту же partition. Consumer-ы объединены в одну consumer group, и Kafka распределяет partitions между ними так, чтобы каждое сообщение внутри группы обрабатывалось одним consumer-ом.

Работа показывает принципы событийной архитектуры, партиционирования сообщений и параллельной обработки событий.

### Лабораторная работа 6. Паттерны web-систем

В директории [`lab_6`](./lab_6) реализован marketplace-сценарий обработки платежных заказов.

Система включает gateway, сервис заказов, сервис платежей, worker, notifier, service discovery, PostgreSQL и Redis. В работе применяются архитектурные и инфраструктурные паттерны: трехзвенная архитектура, CQRS, Pub/Sub, Service Discovery, Heartbeat, Retries с Backoff, Idempotency, Circuit Breaker, Fallback, Graceful Degradation, Backpressure, версионирование и тегирование кэша, ETag, очередь сообщений, polling, long polling и SSE streaming.

Работа отвечает за демонстрацию построения отказоустойчивой распределенной системы, в которой обработка заказов выполняется асинхронно, чтение и запись разделены, а сбои отдельных компонентов обрабатываются контролируемо.

## Запуск лабораторных работ

Каждая лабораторная работа запускается из своей директории.

Общий шаблон запуска:

```bash
cd lab_N
docker compose up --build
```

Для запуска в фоновом режиме:

```bash
docker compose up --build -d
```

Для просмотра логов:

```bash
docker compose logs -f
```

Для остановки контейнеров:

```bash
docker compose down
```

Для остановки с удалением томов, если это предусмотрено конкретной лабораторной работой:

```bash
docker compose down -v
```

## Проверка работ

Подробные команды проверки указаны в `README.md` каждой лабораторной работы. Проверка обычно выполняется через:

- HTTP-запросы `curl` к локальным endpoint-ам;
- просмотр логов контейнеров;
- сравнение ответов сервисов;
- анализ поведения системы при повторных запросах, сбоях или остановке отдельных компонентов.
