# STRIS Labs

Лабораторные работы по дисциплине **Современные технологии разработки информационных систем**.

Репозиторий собран как набор отдельных мини-проектов. Каждая лабораторная лежит в своей папке `lab_N` и показывает отдельную тему: проектирование API, HTTP-сервисы, базы данных, кэширование, репликацию, Kafka, архитектурные паттерны и системный дизайн.

## Как устроен репозиторий

| Папка | Что внутри |
| --- | --- |
| [`lab_0`](./lab_0) | OpenAPI-спецификация REST API социальной сети. |
| [`lab_1`](./lab_1) | Flask-сервис в двух экземплярах, Nginx как reverse proxy и балансировщик, Redis-кэш. |
| [`lab_2`](./lab_2) | PostgreSQL и Redis: индексы, транзакции, уровни изоляции, Cache Aside. |
| [`lab_3`](./lab_3) | Простая in-memory репликация: master и две replica-ноды. |
| [`lab_4`](./lab_4) | Проектирование БД кошелька: партиционирование, шардирование, PostgreSQL-репликация. |
| [`lab_5`](./lab_5) | Kafka: topic, partitions, producer, consumer group и JSON-сообщения. |
| [`lab_6`](./lab_6) | Web-паттерны на примере обработки заказов: gateway, CQRS, discovery, retries, circuit breaker, очереди. |
| [`lab_7`](./lab_7) | Transactional Outbox: запись события в БД и последующая публикация в RabbitMQ. |
| [`lab_8`](./lab_8) | Системный дизайн сервиса уровня Яндекс Такси: нагрузка, память, хранилища, архитектура. |

## Общий запуск

Большинство лабораторных запускается через Docker Compose из своей папки:

```bash
cd lab_N
docker compose up --build
```

Для запуска в фоне:

```bash
docker compose up --build -d
```

Посмотреть логи:

```bash
docker compose logs -f
```

Остановить контейнеры:

```bash
docker compose down
```

Остановить и удалить тома с данными:

```bash
docker compose down -v
```

## Что изучается

В лабораторных показаны основные элементы современных backend- и distributed-систем:

- проектирование REST API и описание контракта через OpenAPI;
- запуск HTTP-сервисов в Docker;
- reverse proxy и балансировка нагрузки через Nginx;
- кэширование через Redis;
- индексы, транзакции и уровни изоляции в PostgreSQL;
- репликация и проблема рассинхронизации данных;
- партиционирование и шардирование;
- событийная обработка через Kafka и RabbitMQ;
- CQRS, service discovery, heartbeat, retry/backoff, circuit breaker и graceful degradation;
- расчет нагрузки, памяти и хранилищ для крупной системы.

## Как проверять

В каждой папке есть свой `README.md` с конкретными командами. Обычно проверка делается через:

- HTTP-запросы `Invoke-RestMethod` или `curl`;
- просмотр логов контейнеров;
- сравнение ответов сервисов;
- остановку отдельных контейнеров для демонстрации сбоев;
- анализ вывода демонстрационного Python-скрипта.

Для начала лучше открыть нужную папку лабораторной и читать локальный `README.md`.
